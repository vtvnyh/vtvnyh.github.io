Real Case: 2018 Uber Crash

01. 坦佩市的深夜

当算法选择了“忽略”

2018年3月18日，Elaine Herzberg 在亚利桑那州被一辆处于自动驾驶模式的 Uber Volvo 撞死。这是全球首例自动驾驶致死案。

调查显示，雷达早在 6秒前 就发现了她，但软件系统将她的分类从“未知”跳变为“车辆”，最后判定为“自行车”。为了避免频繁急刹车影响乘客舒适度（Comfort Mode），系统设定了不立即制动的策略。直到撞击前1.3秒，系统才发出警报，但为时已晚。

核心痛点： 这不是简单的故障，而是工程师在代码中权衡“误报率”与“安全性”时做出的隐性道德选择。

02. 哲学与代码的碰撞

⚖️
功利主义 (Utilitarianism)

核心逻辑： 追求“最大多数人的最大幸福”。

代码表现： if (victims_count > passengers_count) { sacrifice_passenger(); }

困境：如果为了救5个闯红灯的人而牺牲1个遵守规则的乘客，正义何在？

🛡️
义务论 (Deontology)

核心逻辑： 某些行为（如杀害无辜）本身就是错的，无论结果如何。

代码表现： do_not_active_harm(); // 即使这意味着更多人受伤

困境：如果车子只能选择撞墙（死乘客）或撞人（死行人），不作为是否也是一种恶？

🛒
消费者悖论

研究数据： MIT Moral Machine 实验显示，人们普遍赞同“功利主义”的车（救多数人），但自己绝不会买这样的车。

我们希望别人开“道德”的车，但自己想开“安全”的车。

03. CS 视角：损失函数的道德权重

作为开发者，我们并不直接写 kill() 函数。道德困境被转化为概率阈值和损失函数（Loss Function）的权重调整。

  • • 置信度阈值 (Confidence Threshold)： 设多高？太高会漏检（Uber案），太低会幽灵刹车。
  • • 舒适度惩罚 (Comfort Penalty)： 急刹车的负奖励（Negative Reward）设多大？这会间接影响紧急制动的灵敏度。
  • • 长尾问题 (Long-tail Cases)： 训练数据中极少出现的极端场景，模型该如何泛化？
# 简化的决策逻辑伪代码
def decide_action(obstacle, confidence):
    if confidence > 0.9:
        emergency_brake()
    elif confidence > 0.5:
        # 风险：为了舒适性，可能忽略潜在危险
        maintain_speed() 
    else:
        ignore()

# 道德困境：这里的 0.5 阈值由谁决定？

04. 小组圆桌辩论

成员观点碰撞：没有标准答案，只有审慎的思考

A
成员 A (算法方向)

我倾向于功利主义的最优解。虽然冷酷，但从大数据看，如果所有车都遵循“总伤害最小化”，社会总死亡人数会下降。我们可以给不同对象赋予权重，通过计算期望值来做决策。

B
成员 B (系统工程方向)

我反对。技术上很难实时准确识别“儿童”或“老人”，一旦误判就是灾难。而且，如果车企公开承认算法会“主动牺牲乘客”，没人会买车。我建议采用“严格守规”策略：只要车不违规，发生事故由违规方负责，车企不承担道德审判。

C
成员 C (人机交互方向)

问题的核心是信任错位。我认为道德代码不应由程序员在黑盒子里决定。车辆应提供“透明化报告”，甚至在购车时让用户知晓车辆的伦理倾向（虽然这可能被法律禁止）。真正的危机在于我们不知道机器在想什么。

谁来决定生死的代码？

程序员缺乏伦理背景，伦理学家不懂代码实现，大众投票可能情绪化。
我们小组建议：建立独立的算法审计机构，强制安装不可篡改的黑匣子，并制定底线伦理法规（如禁止基于种族/性别的歧视性选择）。

“法律的判决，将决定未来代码的温度。”
